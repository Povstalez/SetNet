# Продуктивність і масштабування SetNet

Цей документ описує: (1) модель продуктивності та як її налаштувати, (2) межі масштабування й захист від перевантаження, (3) структурний roadmap відкладених оптимізацій.

Цифри нижче — з вбудованого бенчмарку (`dotnet run -c Release --project SetNet.Tests -- bench`, in-process loopback, ServerGC), орієнтовні.

---

## 1. Пропускна здатність vs затримка

| Конфігурація | Пропускна здатність (1 з'єднання) | Затримка | Коли використовувати |
|---|---|---|---|
| **Дефолт** (`TcpNoDelay = true`, без батчингу) | ~230k msgs/сек | найнижча (кожне повідомлення йде одразу) | realtime, дрібні рідкі повідомлення, чутливість до затримки |
| **Батчинг** (`SendBatching = true`) | **~1.6M msgs/сек** | низька (один запис на flush) | game-tick, високий темп повідомлень |
| Nagle on (`TcpNoDelay = false`) | ~620k msgs/сек | до ~40 мс (Nagle+delayed-ACK) | масовий потік дрібних повідомлень без вимог до затримки |

**Ключове:** дефолт `TcpNoDelay = true` оптимізує **затримку** — кожне дрібне повідомлення відправляється негайно (без коалесування Nagle), тому «голий» потік незабатчених повідомлень повільніший. Для **високої пропускної здатності вмикайте `SendBatching`**: він коалесує повідомлення тіку в один запис у сокет, даючи і високий throughput, і низьку затримку (~1.6M/сек). Це рекомендований шлях для високого темпу.

```csharp
// Високий темп (game-tick): батчити за тік, флашити раз
var config = new Configuration { SendBatching = true, SendBatchFlushMs = 15 };
// ... за тік:
foreach (var msg in tickMessages) await SendAsync(type, msg);
await FlushAsync(); // один запис у сокет
```

### Пам'ять
~10 КБ на endpoint (in-process, обидва кінці); 2000 з'єднань встановлюються за ~110 мс. Цифри пропускної здатності включають серіалізацію — бібліотека десеріалізує кожне вхідне повідомлення у типізований `T` хендлера.

### Гарячі шляхи (де НЕ алокується зайве)
- **Надсилання:** кадр пишеться в буфер з `ArrayPool`; `SendTimeoutMs` озброює таймер **лениво** — лише якщо `WriteAsync` не завершився синхронно (рідкісний випадок під back-pressure), тож типове надсилання — без алокацій таймера.
- **Reliability-тік:** на «тихому» каналі тік пропускається повністю; коли є unacked — повторно використовується scratch-список (без алокації `List` щотіка).
- **Планувальник:** `TimerScheduler` читає годинник **один раз** за тік (не syscall на кожну реєстрацію).

## 2. Межі масштабування й захист від перевантаження

### Обмежена вхідна черга (`MaxInboundQueue`, дефолт 16384)
Захист від OOM, коли peer надсилає швидше, ніж хендлери споживають:
- **Ненадійні (UDP)** повідомлення понад межу **відкидаються** (best-effort коректно), рахуються в `Metrics.InboundDropped`.
- **Надійний потік** при переповненні **розриває з'єднання** (а не тихо губить впорядковані дані — це краще за corruption чи OOM).
- `0` вимикає межу.

### Що ще обмежено
- `MaxConnectionsLimit` / `MaxUdpPeers` — стелі з'єднань/UDP-peer'ів.
- `MaxMessageSize` — стеля TCP-кадру (slow-loris/OOM).
- `MaxConnectionsPerIpPerSecond` — per-IP rate-limit (з виселенням простоюючих вікон).
- `MaxInFlightMessages` — back-pressure на хендлери (TCP receive-loop призупиняється).

### Відомі стелі (архітектурні)
- **Один UDP-сокет на сервер** + один `_sendLock` серіалізує всі UDP-надсилання всіх peer'ів. Один receive-loop читає всі датаграми. Це стеля пропускної здатності UDP на сервері (див. roadmap S2/S3).
- **`TimerScheduler`** — один цикл O(N) кожні 5 мс; при десятках тисяч реєстрацій варто перейти на timing-wheel (roadmap P3+).
- **Дефолтний dispatch fire-and-forget** (`MaxInFlightMessages = 0`) не обмежує кількість одночасних Task'ів хендлерів — для прод-серверів задавайте >0.

## 3. Структурний roadmap (відкладені оптимізації)

Ці зміни мають хороший потенціал, але зачіпають публічний API або є великим рефакторингом, тож винесені окремо, щоб не дестабілізувати поточну (повністю протестовану) кодову базу. Рекомендований підхід для кожної:

### P4/P5 — Zero-alloc серіалізація і прийом (найбільший важіль для GC)
Зараз кожне повідомлення алокує: `byte[]` від `ISerializer.Serialize` (надсилання) і `byte[]` payload-копію в `PacketBuilder.TryGetCompleteMessage` (прийом).
- **Надсилання:** розширити `ISerializer`/`ITransportConnection.SendAsync` на роботу з `IBufferWriter<byte>`/`ReadOnlyMemory<byte>` у pooled-буфер, передати слайс, повернути буфер після фреймінгу. Прибирає алокацію серіалізації (для MessagePack — через `MessagePackSerializer.Serialize(IBufferWriter<byte>, …)`).
- **Прийом:** для режимів `SequentialDispatch`/`MaxInFlightMessages>0` (де receive-loop відстежує завершення хендлера) — рентити payload з `ArrayPool` і передавати `ReadOnlyMemory<byte>`, повертати після dispatch. Потребує зміни контракту хендлерів (`byte[]` → `ReadOnlyMemory<byte>`).
- **Ризик:** зміна core-API транспорту + контракту хендлерів. Робити як окрему версію (мінорний bump) з міграційним гайдом.

### S2/S3 — Масштабування UDP-сокета
- Прибрати user-space `_sendLock`, використавши `Socket.SendToAsync` з пулом `SocketAsyncEventArgs` (kernel-safe для конкурентних надсилань), і запустити кілька конкурентних `ReceiveAsync` на read-loop'і.
- Альтернатива: шардинг по кількох сокетах (ускладнюється єдиним портом handshake на netstandard2.1 — потрібен SO_REUSEPORT, не портативний).
- **Ризик:** конкурентність на спільному сокеті; ретельне тестування під loss/both.

### S4 — Конкурентний accept + TLS-handshake
Зараз accept і TLS-handshake серіалізовані в одному циклі (DoS закрито таймаутом, але throughput accept обмежений). Рознести: tight accept-loop передає сокет у bounded worker-pool, який робить handshake+`OnNewClient`. **Ризик:** конкурентність, обмеження пулу.

### P3+ — TimerScheduler timing-wheel
Замінити O(N) скан кожні 5 мс на timing-wheel або min-heap за `DueTimestamp`, щоб кожен tick торкався лише прострочених записів. **Ризик:** конкурентність Schedule/Unschedule з структурою (поточний `ConcurrentDictionary` тривіально потокобезпечний).

### S5 — Дефолт back-pressure
Розглянути ненульовий дефолт `MaxInFlightMessages` (або голосне попередження), щоб прод-сервер не плодив необмежено Task'ів за замовчуванням.
